# 비밀번호 XXXXXX

---

## 1. 프로젝트 개요

### 무엇을 만들었는가
암호화된 ZIP 파일(`emergency_storage_key.zip`)의 비밀번호를 브루트포스(완전 탐색)로 크래킹하는 Python 프로그램.

### 핵심 제약 조건
- 비밀번호는 정확히 **6자리**
- 사용 가능 문자: **소문자(a-z) + 숫자(0-9)** → 총 36가지
- 가능한 조합 수: 36^6 = **2,176,782,336 (약 21억 가지)**
- Python 3 표준 라이브러리만 사용 (`zipfile`, `itertools`, `string`, `os`, `zlib`, `datetime`)

---

## 2. 코드 흐름 & 설계 (기본 버전: `door_hacking.py`)

---

### STEP 1 — 파일 유효성 확인

```python
zip_path = 'emergency_storage_key.zip'
output_path = 'password.txt'

# ZIP 파일 존재 여부 확인
if not os.path.exists(zip_path):
    print(f'오류: {zip_path} 파일을 찾을 수 없습니다.')
    return

# ZIP 파일 유효성 확인
try:
    zf = zipfile.ZipFile(zip_path, 'r')
except zipfile.BadZipFile:
    print(f'오류: {zip_path}는 유효하지 않은 zip 파일입니다.')
    return
except OSError as e:
    print(f'파일 열기 오류: {e}')
    return
```

먼저 `os.path.exists()`로 파일 존재 여부를 확인하는 이유는, `zipfile.ZipFile()`을 바로 호출해도 `FileNotFoundError`가 나기는 하지만 오류 메시지가 불친절하기 때문입니다. 사전 확인으로 어떤 문제인지 명확하게 사용자에게 알려줄 수 있습니다.

또한 `BadZipFile` 예외와 `OSError`를 나누어 처리하므로, ZIP 파일 형식이 잘못된 경우와 파일 시스템 오류를 구분하여 대응할 수 있습니다. 이는 나중에 실패 원인을 파악하기 쉽게 만듭니다.

### STEP 2 — 문자셋 & 탐색 공간 설정

```python
# 문자셋: 소문자 + 숫자
charset = string.ascii_lowercase + string.digits
# → 'abcdefghijklmnopqrstuvwxyz0123456789' (36자)
# → 36^6 = 2,176,782,336 (약 21억 가지 조합)

# 시작 시간 기록
start_time = datetime.now()
print(f'시작 시간: {start_time.strftime("%Y-%m-%d %H:%M:%S")}')

attempt_count = 0
found_password = None

# zip 내부 첫 번째 파일 이름 확보 (read() 대상)
first_entry = zf.namelist()[0]
```

`string` 모듈의 상수를 사용하는 이유는 문자열을 하드코딩할 때 발생할 수 있는 오타를 방지하기 위함입니다. 예를 들어 소문자를 입력할 때 'l'(엘)과 '1'(숫자 일)을 실수로 섞을 수 있으므로, 표준 라이브러리의 상수를 사용하는 것이 안전합니다.

`datetime.now()`로 시작 시간을 기록하는 것은 프로그램이 얼마나 오래 실행되었는지 추적하기 위함입니다. 나중에 성능 측정 시 정확한 경과 시간을 계산할 수 있습니다.

`zf.namelist()[0]`은 ZIP 파일 내부의 첫 번째 파일 이름을 미리 꺼내두는 것입니다. 이 파일 이름을 나중에 `read()` 메서드로 비밀번호를 검증할 때 사용합니다. 미리 꺼내두면 21억 번의 반복 루프 내에서 매번 `namelist()`를 호출할 필요가 없어 성능이 향상됩니다.

### STEP 3 — 완전 탐색 루프 (핵심)

```python
# 6자리 모든 조합 생성 및 시도
for combo in itertools.product(charset, repeat=6):
    password = ''.join(combo)
    # 예: ('m','a','r','s','0','6') → 'mars06'
    attempt_count += 1

    # 100만 번마다 진행 상황 출력
    if attempt_count % 1000000 == 0:
        elapsed = datetime.now() - start_time
        print(f'시도 회수: {attempt_count:,} | 경과 시간: {elapsed}')

    try:
        # extractall 대신 read()로 메모리에서만 검증 (디스크 쓰기 없음)
        zf.read(first_entry, pwd=password.encode())
        found_password = password
        break
    except (RuntimeError, zipfile.BadZipFile, zlib.error):
        # 비밀번호 불일치 → 계속 시도
        continue
```

#### 왜 `itertools.product`를 사용하는가?

6중 for 루프로도 동일한 결과를 낼 수 있습니다:
```python
for a in charset:
    for b in charset:
        for c in charset:
            for d in charset:
                for e in charset:
                    for f in charset:
                        password = a + b + c + d + e + f
                        # ... 검증 로직
```

하지만 `itertools.product`를 사용하는 이유는 두 가지입니다:

1. **속도**: `itertools.product`는 C로 구현된 이터레이터라 순수 Python 루프보다 훨씬 빠릅니다.
2. **메모리 효율성**: 제너레이터처럼 동작하여 21억 개의 조합을 모두 메모리에 로드하지 않고, 한 번에 하나씩 생성합니다. 만약 리스트로 만들었다면 메모리 부족으로 프로그램이 크래시할 수 있습니다.

탐색 순서는 사전순이며, **마지막 자리가 가장 빠르게 변합니다**:
```
aaaaaa → aaaaab → aaaaac → ... → aaaaa9 
→ aaaaba → aaaabb → ... → 999999
```

이는 운(luck)에 따라 성능이 크게 달라진다는 의미입니다. 비밀번호가 `aaaaaa`에 가까우면 빠르게 찾고, `999999`에 가까우면 마지막에 찾습니다. 평균적으로는 탐색 공간의 절반을 탐색하고 비밀번호를 찾습니다.

#### 왜 `extractall()` 대신 `read()`인가?

두 메서드의 동작 방식을 비교해보면:

| 메서드 | 동작 | 비밀번호 틀릴 때 | 디스크 I/O |
|--------|------|-----------------|-----------|
| `extractall()` | 파일을 디스크에 실제로 씀 | 빈 파일 생성 후 `zlib.error` | 많음 |
| `read()` | 메모리에서만 복호화 | 즉시 예외, 디스크 I/O 없음 | 없음 |

`extractall()`은 비밀번호를 검증하기 전에 파일부터 만들어놓기 때문에, 21억 번 반복하면 비밀번호가 틀렸을 때도 수많은 빈 파일들이 디스크에 생성됩니다. 이는 디스크 공간을 낭비하고, 디스크 I/O 오버헤드로 인해 속도도 느려집니다.

`read()`는 메모리에서만 복호화를 시도하므로 부작용이 없고 속도도 훨씬 빠릅니다. 성공했을 때만 파일을 추출하거나, 이 프로그램처럼 비밀번호만 저장할 수 있습니다.

#### 왜 예외를 3가지 잡는가?

ZIP의 기본 암호화 방식인 **ZipCrypto (PKWARE Traditional Encryption)**는 비밀번호 검증을 **2단계**로 수행합니다:

**1단계 - 헤더 검증:**
- ZIP 파일의 암호화된 헤더(12바이트)를 복호화합니다.
- 복호화된 헤더의 마지막 1-2바이트를 확인합니다.
- 값이 맞으면 "비밀번호가 맞을 가능성이 높음"이라고 판단합니다.
- 하지만 여기서 문제: **1/256 확률로 틀린 비밀번호도 통과할 수 있습니다** (오탐).
- 실패하면 `RuntimeError` 발생합니다.

**2단계 - 데이터 검증:**
- 1단계를 통과한 후, 실제 DEFLATE 압축 데이터를 복호화합니다.
- 압축 스트림이 유효한지 확인합니다.
- 1단계의 오탐이 발생한 경우, 이 단계에서 압축 스트림이 깨져있어 `zlib.error`가 발생합니다.

따라서 다음과 같이 3가지 예외를 처리해야 합니다:

| 예외 | 발생 시점 | 의미 |
|------|---------|------|
| `RuntimeError` | 1단계 헤더 검증 실패 | 비밀번호 불일치 |
| `zlib.error` | 2단계 데이터 복호화 실패 | 1단계 오탐, 실제로는 비밀번호 불일치 |
| `BadZipFile` | 파일 구조 자체 손상 | ZIP 파일 자체가 손상되었거나, 복구 불가능한 상황 |

어느 단계에서 실패하든 "비밀번호 틀림"이므로 모두 `continue`로 다음 조합을 시도합니다.

#### 100만 번마다 출력하는 이유

매 시도마다 `print()`를 호출하면 I/O 오버헤드가 크래킹 속도를 크게 저하시킵니다. 콘솔에 문자를 쓰는 것도 상당한 비용이 있기 때문입니다. 또한 `datetime.now()` 호출도 시스템 시간을 읽는 비용이 있어 너무 자주 부르면 전체 성능이 떨어집니다.

**100만 번 간격이 적절한 이유:**
- Python 단일 프로세스로 100만 회/초 정도의 속도를 낼 수 있으므로, 약 1초마다 한 번씩 진행 상황을 출력합니다.
- 사용자가 프로그램이 멈추지 않았다는 것을 확인할 수 있을 정도의 주기입니다.
- 너무 많이 출력하지 않으면서도 진행 상황을 추적할 수 있습니다.

### STEP 4 — 결과 저장

```python
zf.close()

elapsed = datetime.now() - start_time

if found_password:
    print(f'\n비밀번호 발견: {found_password}')
    print(f'시도 회수: {attempt_count:,}')
    print(f'경과 시간: {elapsed}')

    # 비밀번호를 password.txt에 저장
    try:
        with open(output_path, 'w') as f:
            f.write(found_password)
        print(f'비밀번호가 {output_path}에 저장되었습니다.')
    except OSError as e:
        print(f'파일 저장 오류: {e}')
else:
    print(f'\n비밀번호를 찾지 못했습니다.')
    print(f'시도 회수: {attempt_count:,}')
    print(f'경과 시간: {elapsed}')
```

비밀번호를 찾은 뒤 ZIP에서 파일을 추출하지 않는 이유는, 이 프로그램의 목적이 비밀번호를 찾는 것이기 때문입니다. 발견된 비밀번호를 `password.txt`에 저장하면 사용자가 직접 ZIP을 열어 필요한 파일들을 확인할 수 있습니다.

파일 저장 시에도 `OSError` 예외를 처리합니다. 디스크 용량 부족, 권한 문제 등으로 파일 쓰기에 실패할 수 있기 때문입니다.

## 3. ZIP 암호화 동작 원리

### ZipCrypto의 2단계 검증

앞서 예외 처리에서 언급했듯이, ZipCrypto는 **스트림 암호** 기반의 암호화입니다.

**구조:**
1. 비밀번호로부터 3개의 32-bit 키를 초기화 (키 스케줄)
2. 파일 데이터의 각 바이트를 XOR 방식으로 암호화
3. 파일 헤더에 **12바이트의 암호화 검증 헤더** 포함
   - 처음 10바이트: 랜덤 데이터
   - 마지막 1-2바이트: 비밀번호 검증용 값

**검증 과정:**
- 12바이트 헤더를 키로 복호화합니다.
- 마지막 1-2바이트를 확인합니다.
- 값이 예상과 일치하면 "비밀번호가 맞을 가능성이 높음"
- 하지만 **256개 중 1개 확률로 틀린 비밀번호도 통과** (오탐)

**오탐 해결:**
- 실제 파일 데이터를 복호화하여 압축 스트림의 유효성 확인
- 압축 스트림이 손상되면 `zlib.error` 발생 → 실제 비밀번호 불일치

## 4. 성능 분석 (기본 버전)

### 탐색 공간 및 예상 시간

- **탐색 공간**: 36^6 = 2,176,782,336 (약 21억)
- **Python 단일 프로세스 속도**: 약 300만~400만 회/초
- **평균 탐색 시간**: 약 9~12분 (탐색 공간의 절반)
- **최악의 경우**: 약 1~2시간 (운이 나쁜 경우)

### 탐색 순서와 운의 영향

`itertools.product`는 사전순으로 탐색하므로, 비밀번호의 위치에 따라 발견 시간이 크게 달라집니다:

```
탐색 순서: aaaaaa → aaaaab → ... → zzzzzz9 → 000000 → ... → 999999
|----------|-----------|-----------|-----------|-----------|
빠름        중간상위   중간        중간하위   느림
(수초)      (분)      (시간)      (시간)    (수시간)
```

비밀번호가 소문자와 숫자가 섞여있으면 중간 어딘가에 위치할 가능성이 높으므로, 평균적으로 탐색 공간의 50%를 탐색합니다.

### 8.1 문제 분석 및 알고리즘 아이디어

기본 버전의 병목은 명확합니다: **단일 프로세스가 21억 가지 조합을 순서대로 하나씩 처리**한다는 것입니다.

현대 컴퓨터는 대부분 **멀티코어 프로세서**를 갖추고 있습니다. 8코어 프로세서면 이론상 최대 8배 빠르게 처리할 수 있지만, 기본 버전은 이를 활용하지 못합니다.

**왜 멀티스레딩이 아닌 멀티프로세싱인가?**

Python의 CPython 구현에는 GIL (Global Interpreter Lock)이라는 메커니즘이 있습니다. GIL은 한 번에 하나의 스레드만 Python 바이트코드를 실행하도록 강제합니다.

- **멀티스레딩**: GIL 때문에 여러 스레드가 있어도 실제로는 한 스레드씩만 실행 → CPU-bound 작업에서는 성능 향상 없음
- **멀티프로세싱**: 각 프로세스가 독립적인 Python 인터프리터를 가지므로 GIL 제약이 없음 → 진정한 병렬 처리 가능

이 프로젝트의 브루트포스는 **CPU-bound 작업**이므로 반드시 **멀티프로세싱**을 사용해야 합니다.

### 8.2 병렬화 전략: 탐색 공간 분할

**핵심 아이디어:**
```
전체 탐색 공간 (36^6)
        │
   첫 글자로 분할
        │
  ┌─────┼─────┐
  ▼     ▼     ▼
 'a'   'b'  ... '9'   ← 각각 36^5 = 6천만 개, 완전히 독립
  │     │         │
  │     │         └─ 프로세스 8: 9aaaaa ~ 9zzzz9
  │     └─ 프로세스 2: baaaaa ~ bzzzz9
  └─ 프로세스 1: aaaaaa ~ azzzz9

(CPU 코어가 8개면, Pool이 8개씩 병렬로 처리)
```

**왜 첫 글자로 분할했는가?**

다른 분할 방식도 가능하지만, 첫 글자 기준이 가장 적절한 이유:

1. **고정 태스크 수**: 첫 글자는 정확히 36가지 → 36개의 완전히 독립적인 태스크
2. **균등 분배**: 각 태스크의 작업량이 정확히 동일 (각 36^5 = 6천만 개)
3. **구현 단순성**: `prefix = 'a'`로 고정하고 나머지 5자리만 순회하면 됨
4. **프로세스 간 통신 최소화**: 태스크들이 완전 독립적이라 공유 데이터 최소화

**다른 분할 방식과의 비교:**

| 분할 방식 | 태스크 수 | 균등성 | 구현 복잡도 | 프로세스 간 통신 |
|---------|---------|--------|----------|-----------------|
| 첫 글자 | 36개 (고정) | 완벽 | 낮음 | 최소 |
| 처음 2글자 | 1,296개 (고정) | 완벽 | 낮음 | 최소 |
| CPU 코어 수만큼 | 8개 (동적) | 부정확 | 높음 (균등 분배 복잡) | 중간 |
| 동적 분배 | 가변 | 정확 | 높음 | 높음 |

첫 글자 분할이 가장 좋은 이유:
- 구현이 간단하면서도 충분히 많은 태스크 (36개)
- 각 태스크의 작업량이 완벽하게 동일
- 추가 분배 로직이 필요 없음

### 8.3 보너스 버전 코드 분석

#### 워커 함수: `_crack_worker(args)`

```python
def _crack_worker(args):
    # 각 프로세스에서 실행: 첫 글자를 고정하고 나머지 5자리를 완전 탐색
    zip_path, prefix, charset, stop_event, result_queue = args

    try:
        # 각 프로세스가 독립적으로 ZIP 파일을 열어야 함
        # (ZipFile 객체는 프로세스 간 공유 불가능)
        zf = zipfile.ZipFile(zip_path, 'r')
        first_entry = zf.namelist()[0]

        # prefix를 고정하고 나머지 5자리만 순회
        for combo in itertools.product(charset, repeat=5):
            # 다른 프로세스가 이미 비밀번호를 찾았으면 즉시 종료
            if stop_event.is_set():
                zf.close()
                return

            # 첫 글자(prefix) + 나머지 5자리
            password = prefix + ''.join(combo)
            
            try:
                # 기본 버전과 동일한 검증 로직
                zf.read(first_entry, pwd=password.encode())
                
                # 비밀번호 발견!
                result_queue.put(password)      # 메인 프로세스로 결과 전달
                stop_event.set()                # 나머지 프로세스에 종료 신호
                zf.close()
                return
            except (RuntimeError, zipfile.BadZipFile, zlib.error):
                # 비밀번호 불일치 → 다음 조합 시도
                continue
        
        # 이 프로세스의 담당 구간에서 찾지 못함
        zf.close()
    except Exception:
        # 워커 내부 오류 발생 시 다른 프로세스에 영향 없이 조용히 종료
        # (예: 파일 읽기 권한 오류, 메모리 부족 등)
        pass
```

**워커 함수의 핵심 설계:**

1. **독립적인 파일 열기**: 각 프로세스가 ZIP 파일을 별도로 열어야 합니다. `ZipFile` 객체는 내부 상태(파일 포인터, 버퍼 등)를 가지므로 프로세스 간 공유가 불가능합니다.

2. **`stop_event` 모니터링**: 매 반복마다 `if stop_event.is_set():`을 확인합니다. 다른 프로세스가 비밀번호를 찾으면 `stop_event`를 설정하므로, 이를 감지한 프로세스들이 조기 종료할 수 있습니다.

3. **결과 전달 (`result_queue`)**: 비밀번호를 찾으면 큐에 넣어 메인 프로세스로 전달합니다. 프로세스 간 데이터 통신은 큐를 통해 안전하게 이루어집니다.

4. **예외 처리**: 워커 내부에서 발생하는 모든 예외를 최상위 `except Exception:`로 잡습니다. 이렇게 하면 한 워커의 오류가 다른 워커들에 영향을 주지 않습니다. (대신 조용히 실패하므로 메인 프로세스에서 `result_queue` 상태를 확인해야 합니다.)

#### 메인 함수: `unlock_zip_fast()`

```python
def unlock_zip_fast():
    zip_path = 'emergency_storage_key.zip'
    output_path = 'password.txt'

    # 기본 버전과 동일한 파일 검증
    if not os.path.exists(zip_path):
        print(f'오류: {zip_path} 파일을 찾을 수 없습니다.')
        return

    try:
        zf_check = zipfile.ZipFile(zip_path, 'r')
        zf_check.close()
    except zipfile.BadZipFile:
        print(f'오류: {zip_path}는 유효하지 않은 zip 파일입니다.')
        return
    except OSError as e:
        print(f'파일 열기 오류: {e}')
        return

    # 문자셋: 소문자 + 숫자
    charset = string.ascii_lowercase + string.digits
    
    # 현재 시스템의 CPU 코어 수 감지
    cpu_count = multiprocessing.cpu_count()

    # 시작 정보 출력
    start_time = datetime.now()
    print(f'시작 시간: {start_time.strftime("%Y-%m-%d %H:%M:%S")}')
    print(f'사용 CPU 코어 수: {cpu_count}')
    print(f'탐색 공간: {len(charset)}^6 = {len(charset) ** 6:,}')
    print(f'프로세스당 탐색 범위: {len(charset) ** 5:,} (첫 글자 1개 고정)')

    # 프로세스 간 공유 객체 생성 (Manager 사용)
    manager = multiprocessing.Manager()
    stop_event = manager.Event()               # 프로세스 간 공유 이벤트 플래그
    result_queue = manager.Queue()             # 발견된 비밀번호 전달용 큐

    # 36개 태스크 생성: 각 태스크는 첫 글자 1개와 나머지 5자리
    tasks = [
        (zip_path, char, charset, stop_event, result_queue)
        for char in charset
    ]
    # tasks = [
    #     (zip_path, 'a', charset, stop_event, result_queue),
    #     (zip_path, 'b', charset, stop_event, result_queue),
    #     ...
    #     (zip_path, '9', charset, stop_event, result_queue),
    # ]

    # 멀티프로세싱 Pool로 병렬 실행
    # Pool은 자동으로 cpu_count만큼의 워커 프로세스를 생성
    # 36개 태스크를 cpu_count개씩 분배해 순차적으로 처리
    with multiprocessing.Pool(processes=cpu_count) as pool:
        # pool.map()은 모든 태스크가 완료될 때까지 대기
        # 반환값이 있으면 리스트로 반환하지만, 워커는 반환값이 없으므로 무시
        pool.map(_crack_worker, tasks)

    # 결과 추출
    elapsed = datetime.now() - start_time
    found_password = None

    if not result_queue.empty():
        found_password = result_queue.get()

    # 기본 버전과 동일한 결과 처리
    if found_password:
        print(f'\n비밀번호 발견: {found_password}')
        print(f'경과 시간: {elapsed}')

        try:
            with open(output_path, 'w') as f:
                f.write(found_password)
            print(f'비밀번호가 {output_path}에 저장되었습니다.')
        except OSError as e:
            print(f'파일 저장 오류: {e}')
    else:
        print(f'\n비밀번호를 찾지 못했습니다.')
        print(f'경과 시간: {elapsed}')
```

**메인 함수의 핵심 설계:**

1. **`Manager()` 사용 이유**: 
   - 일반 `multiprocessing.Event()`는 Linux의 `fork` 방식에서만 안정적으로 프로세스 간 공유가 되지만, macOS/Windows의 `spawn` 방식에서는 보장되지 않습니다.
   - `Manager`는 별도의 서버 프로세스를 띄워 Event와 Queue를 관리하므로, 모든 OS에서 안전하게 프로세스 간 공유가 가능합니다.

2. **`stop_event` 없으면 어떻게 되는가?**
   - 예를 들어, 비밀번호의 첫 글자가 `c`인 경우, `c` 워커가 빨리 찾을 것입니다.
   - 하지만 `stop_event`가 없으면 나머지 35개 워커(`a`, `b`, `d~9`)는 자신의 6천만 번 탐색을 **끝까지 마치고 나서야** 종료됩니다.
   - 결과적으로 비밀번호를 찾고도 한 시간을 더 기다려야 합니다!
   - `stop_event.is_set()` 덕분에 다른 워커들이 다음 반복에서 즉시 종료할 수 있습니다.

3. **`pool.map()` 동작:**
   - `pool.map(func, iterable)`은 `iterable`의 각 요소에 `func`을 적용합니다.
   - 내부적으로 `cpu_count`개의 워커 프로세스를 유지하면서, 36개 태스크를 순차적으로 분배합니다.
   - 모든 태스크가 완료될 때까지 블로킹됩니다.

4. **CPU 코어 수 감지:**
   - `multiprocessing.cpu_count()`로 현재 시스템의 CPU 코어 수를 자동 감지합니다.
   - 예: 8코어 → 최대 8개 워커 병렬 실행
   - 4코어 → 최대 4개 워커 병렬 실행
   - 시스템 자원에 맞게 자동 조정됩니다.

1. **`if __name__ == '__main__':`의 중요성**:
   - 이 코드 구조는 `multiprocessing`이 안정적으로 작동하기 위해 **필수**입니다.
   - 보너스 버전에서 이 구문이 없으면, 워커 프로세스 생성 시 모듈을 임포트하면서 `unlock_zip_fast()`가 재귀적으로 호출되어 무한 루프나 크래시 발생 가능
   - 기본 버전에서도 재사용 가능한 모듈로 만들 때 필요

2. **ZIP 파일 보안**:
   - ZipCrypto는 1989년에 설계된 스트림 암호로, 현대 암호화 기준으로는 취약합니다.
   - 민감한 정보는 7-Zip의 AES-256 암호화 등 현대적 방식 사용 권장

3. **파일 시스템 용량**:
   - 기본 버전은 디스크에 파일을 쓰지 않으므로 안전
   - ZIP 파일을 실제로 추출할 때는 충분한 디스크 공간 확인
